Plotting & saving solutions

Once solve hands you a PDESolution, two things you almost always want next: see it and keep it. This chapter covers plot_solution, Makie for 2-D/3-D views, and an opt-in run-history recorder that streams every snapshot to disk in a plain, language-agnostic format.

1 · Plotting a solution

The package ships plot_solution(sol), which dispatches on the problem type stored in sol.problem and returns the standard Plots.jl figure for it. You can also just call plot(sol) — the package extends Plots.plot for both problem structs and solutions.

julia — one call per problem type
using juliaPDEs, Plots

# 1-D heat → snapshots + space-time heatmap
grid = Grid(a = (0.0,), b = (1.0,), stepsize = (1/200,))
tg   = TestGrid(grid, (t -> 0.0,), (t -> 0.0,), x -> exp(-100 * (x - 0.5)^2))
sol  = solve(HeatEquation(testgrid = tg, Nt = 2000, T = 1.0, α = 0.01))

plot_solution(sol)                 # or simply: plot(sol)
plot_solution(sol; savepath = "heat.png")   # also write a PNG

# 2-D Poisson → f(x,y) / u(x,y) / 3-D surface
plot(solve(PoissonEquation()))

2-D & 3-D views with Makie

For surfaces, filled contours and genuinely 3-D fields, Makie is the sharper tool. The package depends on CairoMakie (static PNG/SVG, headless) and GLMakie (interactive). Because a PDESolution exposes sol.x, sol.y, sol.z and sol.u, you feed it straight to Makie:

julia — Makie 2-D heatmap, contour & 3-D surface
using juliaPDEs, CairoMakie     # CairoMakie ⇒ static, no GPU needed

sol = solve(PoissonEquation())  # 2-D, sol.u is a matrix

# Heatmap
fig = Figure(); ax = Axis(fig[1,1]; xlabel="x", ylabel="y", aspect=DataAspect())
hm  = heatmap!(ax, sol.x, sol.y, sol.u; colormap=:viridis)
Colorbar(fig[1,2], hm); save("heatmap.png", fig)

# Filled contour + iso-lines
levels = range(minimum(sol.u), maximum(sol.u); length=12)
fig = Figure(); ax = Axis(fig[1,1]; aspect=DataAspect())
contourf!(ax, sol.x, sol.y, sol.u; levels, colormap=:viridis)
contour!( ax, sol.x, sol.y, sol.u; levels, color=:black, linewidth=0.9)
save("contour.png", fig)

# 3-D surface
fig = Figure(); ax = Axis3(fig[1,1]; xlabel="x", ylabel="y", zlabel="u")
surface!(ax, sol.x, sol.y, sol.u; colormap=:viridis)
save("surface.png", fig)
julia — a 3-D field as stacked z-slices
# For a 3-D solution (sol.u is a 3-D array), draw a few z-slices as colored
# surface planes — CairoMakie renders this without a GPU. (True iso-surfaces /
# volume rendering need GLMakie.)
fig = Figure(); ax = Axis3(fig[1,1]; xlabel="x", ylabel="y", zlabel="z")
lo, hi = extrema(sol.u)
for frac in (0.25, 0.5, 0.75)
    k      = round(Int, frac * length(sol.z))
    zplane = fill(sol.z[k], length(sol.x), length(sol.y))
    surface!(ax, sol.x, sol.y, zplane;
             color = sol.u[:, :, k], colormap = :viridis,
             colorrange = (lo, hi), transparency = true)
end
save("slices.png", fig)
Backends CairoMakie writes PNG/SVG/PDF and runs headless on a cluster; GLMakie opens an interactive window (needs OpenGL / a display) and is the one to load for rotatable iso-surfaces and volume rendering.

2 · Saving a run's history

For time-dependent problems you often want the whole evolution, not just the final frame. Pass save_every (and optionally save_dir) to solve — supported by HeatEquation and WaveEquation — and each snapshot is written as it is computed. Omit save_every and the solver takes its original fast path: nothing is saved.

julia — record while solving
grid = Grid(a = (0.0,), b = (1.0,), stepsize = (1/50,))
tg   = TestGrid(grid, (t -> 0.0,), (t -> 0.0,), x -> exp(-100 * (x - 0.5)^2))
prob = HeatEquation(testgrid = tg, Nt = 100, T = 0.2, α = 0.01)

# Save the field every 10 iterations (plus the initial and final states)
sol = solve(prob; save_every = 10)
#   ⇒ runs/heatequation_1d_N50_2026-06-17T08-28-53/

# Choose the folder yourself:
sol = solve(prob; save_every = 10, save_dir = "runs/my_experiment")

# Just the final solution? One frame:
save_solution(sol, "runs/final_only")

Each run lands in its own descriptive, timestamped folder. All metadata lives once in meta.json; the per-iteration files hold nothing but the field values, so they stay tiny:

run folder layout
runs/heatequation_1d_N50_2026-06-17T08-28-53/
├── meta.json          # problem, params, grid, dt, shape + index of every frame
├── step_00000.json    # field values ONLY — flat, column-major, no metadata
├── step_00001.json
├── step_00002.json
└── ...
json — meta.json (abridged)
{
  "problem": "HeatEquation",
  "dimension": 1,
  "dt": 0.002,
  "shape": [50],
  "order": "column-major",
  "grid": { "x": [0.0196, 0.0392, "..."] },
  "params": { "Nt": 100, "T": 0.2, "α": 0.01 },
  "snapshots": [
    { "step": 0, "iteration": 0,  "t": 0.0,  "file": "step_00000.json" },
    { "step": 1, "iteration": 10, "t": 0.02, "file": "step_00001.json" }
  ]
}
Why metadata in one place? Repeating the grid and parameters in every frame would multiply the disk cost of a long run for no benefit. Keeping them once in meta.json — and storing each frame as a bare JSON number array — keeps the run small and readable by any language.

3 · Reloading — in Julia or anywhere else

Because the format is plain JSON, you can replay a run in Julia or pull it into Python/MATLAB for post-processing. The flat frames reshape back with meta["shape"] in column-major (Fortran) order.

julia — load_history
meta, times, frames = load_history("runs/my_experiment")

length(frames)        # number of saved snapshots
times                 # Vector of simulation times, one per frame
frames[end]           # final field, reshaped to meta["shape"]
python — same run, no Julia needed
import json, numpy as np, os

run   = "runs/my_experiment"
meta  = json.load(open(os.path.join(run, "meta.json")))
shape = meta["shape"]

frames = []
for snap in meta["snapshots"]:
    flat = json.load(open(os.path.join(run, snap["file"])))
    frames.append(np.array(flat).reshape(shape, order="F"))  # column-major

times = [s["t"] for s in meta["snapshots"]]
