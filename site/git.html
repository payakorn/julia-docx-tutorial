Chapter 15

Git & GitHub

The minimum Git you need to share your code and collaborate on a project.

Share your code with Git & GitHub

You now have a working Julia project with PDE solvers and an HPC job script. The last step is to put your code under version control with Git, and push a copy to GitHub so you can access it from anywhere (including the cluster) and share it with classmates or your advisor.

What is Git, in one sentence? Git records snapshots of your project over time so you can see what changed, undo mistakes, and collaborate without emailing zip files. GitHub is a website that hosts those snapshots online.

1. Install Git (inside WSL)

Open your Ubuntu terminal and run:

bash
sudo apt update
sudo apt install -y git
git --version

The last command should print something like git version 2.43.0. That confirms Git is installed.

2. Tell Git who you are (one-time setup)

Every snapshot is stamped with your name and email. Set them once and Git remembers them for every project on this computer:

bash
git config --global user.name  "Your Name"
git config --global user.email "you@example.com"

Use the same email as your GitHub account when you create one in step 6.

3. Initialize the repo in your project folder

Go to the folder that holds your Julia project (the one with Project.toml) and turn it into a Git repository:

bash
cd ~/projects/julia-docx-tutorial   # adjust to your folder
git init
git status

git init creates a hidden .git/ folder where Git stores all its history. git status shows every file Git can see; right now they will all be listed as untracked.

4. Tell Git which files to ignore

You do not want to commit generated PNG figures, the large .docx, Julia’s package cache, or editor temp files. Create a .gitignore file at the top of the project with this content:

bash
# Julia
Manifest.toml
*.jl.cov
*.jl.mem
.julia/

# Generated artifacts
output/
site/output/figures/
*.docx
*.png

# OS / editor
.DS_Store
.vscode/
*.swp

Each line is a pattern. Anything matching is hidden from Git. You can always remove a line later if you decide you do want to track that file.

5. Make your first commit

A commit is a labelled snapshot of every file currently staged. Stage all files, then commit with a short message:

bash
git add .
git status                          # confirm what is staged
git commit -m "Initial project: Julia PDE solvers and setup guide"

Git replies with something like [main (root-commit) 9e4c42c] Initial project .... That hash is the snapshot’s unique ID; you can always come back to it later.

Good commit messages Write what you did and why, in the present tense: “Add heat-equation solver and convergence test” is better than “changes” or “wip”.

6. Create a GitHub account and an empty repository

  1. Go to github.com and sign up (free).
  2. Click + → New repository in the top-right.
  3. Name it (e.g. julia-pde-tutorial), keep it public or private as you prefer, and do not tick “Add a README” or “Add .gitignore” — your local folder already has those.
  4. Click Create repository. GitHub shows a page with the URL of your new (empty) repo.

7. Connect your local repo to GitHub and push

On the GitHub repo page, copy the HTTPS URL (looks like https://github.com/<you>/julia-pde-tutorial.git). Then in your terminal:

bash
git branch -M main
git remote add origin https://github.com/<your-username>/julia-pde-tutorial.git
git push -u origin main

The first time you push, GitHub will ask for your username and a personal access token (PAT) instead of a password. Create one at github.com/settings/tokens → Generate new token (classic), give it the repo scope, and paste it where Git asks for the password. Git will cache it so you don’t have to enter it again.

8. Day-to-day loop

From now on, every time you change something you want to keep:

bash
git status                          # what changed?
git add path/to/file.jl             # stage specific files (or use "git add ." for all)
git commit -m "Fix CFL bug in wave solver"
git push                            # send the new commits to GitHub

9. Stop typing tokens: set up an SSH key (public & private)

Typing a personal access token on every push gets old fast. The cleaner way is an SSH key pair. Think of it as a lock and a key:

  • the private key (id_ed25519) stays secret on your machine — never share it, never commit it;
  • the public key (id_ed25519.pub) is the matching lock — you hand a copy to GitHub. Anyone can have it.

When you push, your machine proves it holds the private key that matches the public lock GitHub has on file. No password, no token.

a) Generate the key pair (use your GitHub email):

bash
ssh-keygen -t ed25519 -C "you@example.com"

Press Enter three times to accept the default location (~/.ssh/id_ed25519) and an empty passphrase. You now have two files in ~/.ssh/: id_ed25519 (private — keep it) and id_ed25519.pub (public — share it).

Never share the private key. Only the file ending in .pub ever leaves your machine. If someone gets your private key they can push as you — and that is exactly why .ssh/ is never inside a Git project folder.

b) Copy the public key to your clipboard — print it and copy the output:

bash
cat ~/.ssh/id_ed25519.pub

It is one line starting with ssh-ed25519 and ending with your email. Select the whole line and copy it.

c) Add it to GitHub: go to github.com/settings/keys → New SSH key, give it a title (e.g. “WSL laptop”), paste the public key, and click Add SSH key.

d) Test it and switch your remote to SSH:

bash
ssh -T git@github.com          # should greet you by username
git remote set-url origin git@github.com:<your-username>/julia-pde-tutorial.git
git push                       # no more token prompts

The first ssh -T asks “Are you sure you want to continue connecting?” — type yes. After that, pushing and pulling just work. Note the SSH URL uses git@github.com: instead of https://.

10. Add a collaborator (co-author) to your repo

To let a classmate or advisor push and edit code in the same repository, add them as a collaborator. They get their own clone and push directly — no zip files, no copy-paste.

  1. On your repo page, open Settings → Collaborators (you may be asked to confirm your password).
  2. Click Add people and type their GitHub username or email.
  3. Pick a role — Write lets them push commits, which is what you want for a co-author.
  4. GitHub emails them an invitation. Once they accept, they clone the repo and work normally:
bash
git clone git@github.com:<your-username>/julia-pde-tutorial.git
cd julia-pde-tutorial
# ...edit files...
git add .
git commit -m "Add Crank–Nicolson option to heat solver"
git push
Credit a true co-author on a single commit. If two of you wrote one change together, add a trailer to the commit message so GitHub shows both faces on the commit:
bash
git commit -m "Add lid-driven cavity benchmark

Co-authored-by: Class Mate <mate@example.com>"

11. Work on a branch (don’t break main)

A branch is a parallel line of work. You and your co-author can each experiment on your own branch without touching the working code on main; when a feature is ready, you merge it back. This is the heart of safe collaboration.

a) Create a branch and switch to it:

bash
git switch -c feature/wave-3d      # create + switch in one step
git branch                          # the * marks your current branch

Commit on this branch as usual. The first time you push it, tell GitHub to track it:

bash
git add .
git commit -m "Add 3D leap-frog wave solver"
git push -u origin feature/wave-3d

b) Open a Pull Request. Refresh your repo on GitHub — it shows a Compare & pull request button. A Pull Request (PR) proposes merging your branch into main. Your co-author can review the diff, comment line by line, and approve. Click Merge pull request when ready.

c) Bring your local copy up to date after a merge:

bash
git switch main                     # go back to the main branch
git pull                            # download merged changes from GitHub
git branch -d feature/wave-3d       # delete the finished branch (local)
The golden rule of branches. Always git pull on main before starting a new branch, so you build on your co-author’s latest work. Keep branches small and short-lived — one feature or fix each — and they stay easy to review and merge.
You’re done. Refresh your GitHub repo page in the browser — you should see every file from your project, your commit message at the top, and a clickable history. From here you can git clone the same repo on the HPC cluster and run the exact same code there.
